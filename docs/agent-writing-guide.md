# 記事執筆ガイド

LLM書庫の記事を書く・編集するときの文体、Markdown整理、対話、作業後報告のルールです。

## LLM書庫の文体

記事は、LLM書庫の文体で書きます。

- ロジカルで簡潔に書く
- 詩的にしすぎない
- 思想を大きく語りすぎない
- 同じ説明を繰り返さない
- LLMでリサーチ・要約・整理した知識片を、あとから読める形で残すための記事として書く
- AIはリサーチ・要約・構成・執筆を行う存在として扱う
- 人間はテーマ設定、公開判断、残す価値の判断を行う
- 見出しは少なく、本文は短くする
- 不確かなことは不確かなまま書く
- 600〜1000字程度に収める

## Markdownの読みやすさ

記事は、プレーンな本文だけで詰め込みすぎず、Markdownとして読みやすい形に整えます。

- 重要な主張、記事の結論、読者に残したい言葉は **太字** にする
- ユーザーが指定した言葉、引用したい短い表現、記事の核になる一文は引用ブロックや太字で目立たせる
- 複数の要素を並べるときは、本文に埋め込まず、箇条書きや番号付きリストを使う
- 条件、例外、比較、判断手順は、必要に応じてネストしたリストで整理する
- 専門用語や本文理解に必要な言葉は、短い用語メモとして置く
- 用語メモ内の重要語、ユーザーが明示した語、記事の中心概念は **太字** にする
- 太字やリストは読みやすさのために使い、記事を百科事典的・レポート的に広げすぎない
- 本文を書いたあと、段落だけで読みにくい箇所を見直し、太字、引用、リスト、用語メモで再構成できないか試行錯誤する

## 記事作成時の対話ルール

記事を作成するとき、AI AgentはユーザーにCLI上で選択を求めません。

frontmatter は本文と依頼内容から Codex が仮入力します。
判断に迷う場合は、次の初期値にします。

```yaml
status: seed
confidence: medium
review_needed: true
```

ユーザーに確認が必要な場合も、CLIの選択肢ではなく自然文で確認します。

例: 「このノートはまだ仮説扱いにしています。確定寄りにする場合は status を growing に変更できます。」

## 作業後の報告ルール

新しい記事を追加した場合は、作業後の報告で必ず Starlight の左サイドバー運用について提案します。

既存カテゴリに入れるならカテゴリ名を示し、新しいカテゴリが必要そうなら理由と候補名を自然文で提案します。

ユーザーから明示されていない限り、新カテゴリの追加やサイドバー再編は勝手に大きく行いません。

作業後、PRコメントまたは最終報告で、frontmatter の意味を日本語で説明します。

例:

- 状態: 🌱 `seed`（未整理の断片、仮説、下書き）
- 情報源: `web`（Web記事由来）
- AI処理: `summary` / `structure`（要約と構造化）
- 確信度: `medium`（中。未確認の部分がある）
- 見直し: 必要
