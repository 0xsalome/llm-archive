# frontmatter ガイド

LLM書庫の記事では、experimental-commons の provenance / knowledge_status の思想を引き継ぎます。

ただし、非エンジニアでも運用できるように、frontmatterは軽量にします。

## 軽量frontmatterの基本形

```yaml
---
title: 記事タイトル
description: 1〜2文の説明
date: 2026-07-06
status: seed
tags:
  - AI
  - 運用
source_type: ai_session
source_ref: "ChatGPT session, 2026-07-06"
ai_process:
  - structure
  - rewrite
confidence: medium
review_needed: true
related_notes: []
---
```

## title

記事のタイトルです。

読者が一覧で見たときに、中身が分かる名前にします。

主題と副題を分ける長いタイトルでは、区切りに `——` を使います。
トップページの記事カードと記事ページでは、`——` 以降を小さめのサブタイトルとして改行表示します。
`――` は既存記事の互換として表示側で読み取りますが、新しく書く記事では `——` に統一します。

## description

1〜2文の説明です。

「何についてのノートか」「なぜ残すのか」が分かるようにします。

## date

その知識を公開・更新した日です。

迷ったら、記事を作った日を入れます。

## status

公開上の成熟度です。

| status | 意味 |
| --- | --- |
| 🌱 `seed` | 未整理の断片、仮説、下書き |
| 🌿 `growing` | 整理中で、ある程度読める |
| 🔎 `checked` | 出典や論理を一度確認した段階 |
| 🧭 `superseded` | 後のノートによって見方が更新された段階 |

迷ったら `seed` にします。

`checked` は「永遠に正しい」ではありません。現時点で出典や論理を一度確認した、という意味です。
見方が後のノートで更新された場合は、古いノートを消さずに `superseded` として残します。

## tags

探しやすくするためのキーワードです。

最初は少なくて構いません。

おすすめのタグ:

- `技術`
- `データ`
- `読書`
- `観察`
- `環境`
- `科学`
- `AI`
- `制作`
- `運用`

## タグ運用

タグは、トップページで記事を絞り込むための軽いカテゴリーとして扱います。

- 1つの記事につきタグは2〜3個にする
- 記事の中心テーマを表すタグだけを残す
- 似た意味のタグをむやみに増やさない
- 新しいタグを作る前に、既存タグで表せないか確認する
- 固有名詞だけのタグより、あとから複数記事を束ねられるタグを優先する
- タグは記事タイトルの要約ではなく、複数記事を束ねるための固定語彙として扱う
- 調査記事では、まず `技術` / `データ` / `読書` / `観察` / `AI` / `制作` / `運用` の既存系統で表せるか確認する
- 1記事のためだけのタグは原則作らない

## 軽量メタデータ

出典、AI処理、確信度、レビュー要否、関連ノートは、機械的に管理しやすい最低限の情報としてfrontmatterにも残します。

分からない項目は無理に埋めず、本文に「要確認」と書いて構いません。

## source_type

知識の出どころです。

使える値:

- `personal_note`
- `ai_session`
- `web_research`
- `book`
- `paper`
- `official_source`
- `other`

## source_ref

後から辿れる手がかりです。

例:

- `ChatGPT session, 2026-07-06`
- `個人メモ: LLM書庫設計`
- `書籍名、章、ページ`
- `URL`

秘密情報や非公開情報は書きません。

## ai_process

AIが何をしたかを短く残します。

使える値:

- `draft`
- `summarize`
- `extract`
- `structure`
- `rewrite`
- `compare`
- `fact_check`
- `critique`

複数ある場合は配列で書きます。

```yaml
ai_process:
  - extract
  - structure
  - rewrite
```

## confidence

現時点の確信度です。

| value | 意味 |
| --- | --- |
| `low` | 出典が弱い、観察が少ない、仮説段階 |
| `medium` | ある程度根拠はあるが、まだ要検証 |
| `high` | 一次情報源や十分な確認がある |

迷ったら `medium` 以下にします。

## review_needed

人間の再確認が必要なら `true` にします。

迷ったら `true` にします。

## related_notes

関連する既存ノートがあれば入れます。

まだなければ空配列で構いません。

```yaml
related_notes:
  - /archive/first-note/
```

## frontmatterに残すこと

- `source_type`: 知識の出どころ
- `source_ref`: 後から辿れる手がかり
- `ai_process`: AIが行った処理
- `confidence`: 現時点の確信度
- `review_needed`: 人間レビューが必要か
- `related_notes`: 関連ノート

## 本文に残すこと

frontmatterは機械的に管理するための最低限の情報です。

人間が読み返すための文脈は本文にも書きます。

本文に残すもの:

- 何を観察したか
- まだ確信がない点
- どこから来た情報か
- なぜその出典を使ったか
- AIが何をしたか、どこまで手伝ったか
- 古いノートとどう関係するのか
- 何を次に見直すのか

## 次に残る問い

各記事の結論後には、必要に応じて3行程度の「次に残る問い」を置きます。

ここでは、未確認の点、結論の射程、今後調べるべき問いなどを短く残します。
記事を確定知識のように閉じず、調査途中の余白を保つための小さな保留として扱います。

## 古い内容の扱い

既存ノートを黙って削除・上書きしません。

見解が変わった場合は、古いノートを消すのではなく、更新メモ、関連ノート、比較ノートとして残します。

矛盾が見つかった場合は、どちらかを即座に正解とせず、レビュー候補として扱います。
