# カテゴリと内部リンクの考え方

LLM書庫では、最初から細かいカテゴリやタグを作りすぎません。

記事が増えるまでは、Starlightの左サイドバー、少数のタグ、内部リンクでゆるくつなぎます。

カテゴリは読者が左サイドバーから辿るための大きな棚です。
タグはトップページで記事を絞り込むための軽い分類です。
タグを記事ごとに増やすと一覧が読みにくくなるため、固定語彙に近いものとして扱います。

## 最初のカテゴリ案

| カテゴリ | 置くもの |
| --- | --- |
| `Start Here` | この書庫の前提、読み方、運用思想 |
| `Technology` | 技術、データ、AI、日常にあるシステムの観察 |
| `Ecology` | 環境、人文学、生態、自然観察、環境思想 |
| `Reading` | 書籍、論文、記事、動画のメモ |
| `Observations` | 個人的な観察、違和感、発見 |
| `Projects` | 制作物や実験の記録 |
| `Operations` | 書き方、調査、編集、公開の手順 |
| `References` | ライセンス、スキーマ、用語整理、参照用の基礎ノート |

カテゴリは `astro.config.mjs` の Starlight `sidebar` で管理します。
記事を追加しただけでは、左サイドバーに自動では出ません。
新しい記事を読者に常時見せたい場合は、該当カテゴリの `items` に追加します。

実際のファイルは、当面 `src/content/docs/archive/` に平らに置いて構いません。
記事数が増えてきたら、必要に応じて次のように分けます。

```text
src/content/docs/
  technology/
  reading/
  observations/
  projects/
  operations/
  references/
```

## タグの考え方

タグは増やしすぎません。
記事ごとの細かい話題ではなく、複数記事を束ねられる語を使います。

最初は次の系統を優先します。

- `技術`
- `データ`
- `読書`
- `観察`
- `環境`
- `科学`
- `AI`
- `制作`
- `運用`

例:

```yaml
tags:
  - 技術
  - データ
```

固有名詞、個別サービス名、1記事だけで終わる語は、タグではなく本文や出典欄に置きます。

## 内部リンクの付け方

本文の最後に「関連ノート」を置きます。

```md
## 関連ノート

- [LLM書庫を始める](/llm-archive/archive/first-note/)
- まだ記事がなければ「なし」
```

## リンク候補を残すだけでもよい

まだ記事がない場合は、無理にリンクを作りません。

```md
## 関連ノート

- 今後作りたい: LLMとの読書メモの作り方
- 今後作りたい: AIが書いた文章を人間がレビューする手順
```

## 古いノートと違う考えになったとき

古いノートを黙って消さず、次のどれかをします。

- 新しいノートから古いノートへリンクする
- 古いノートに「更新メモ」を足す
- 比較ノートを作る
- 「この考えは変わった」と本文に残す

知識が変わること自体は失敗ではありません。

変わった跡を残すことが、この書庫の価値になります。
