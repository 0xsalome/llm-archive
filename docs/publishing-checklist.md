# 公開前チェックリスト

記事をPull Requestに出す前、または公開前に確認します。

全部を完璧に満たす必要はありません。

不完全なら、本文に「要検証」「まだ確信がない」と書けば公開候補にできます。

## frontmatter

- [ ] `title` が入っている
- [ ] 長いタイトルで副題を付ける場合、区切りは `——` に統一している
- [ ] `description` が1〜2文で入っている
- [ ] `date` が `YYYY-MM-DD` 形式になっている
- [ ] `status` が 🌱 `seed` / 🌿 `growing` / 🔎 `checked` / 🧭 `superseded` のどれかになっている
- [ ] `tags` が1つ以上入っている
- [ ] `source_type` と `source_ref` で、出どころを短く残している
- [ ] AIが関与した場合は `ai_process` に処理内容を残している
- [ ] `confidence` が `low` / `medium` / `high` のどれかになっている
- [ ] 人間レビューが必要なら `review_needed: true` になっている
- [ ] 関連ノートがあれば `related_notes` に入っている

## 本文

- [ ] 何を観察したかが書かれている
- [ ] 仮説と事実が混ざりすぎていない
- [ ] 不確かなことに「要検証」「現時点では」などが付いている
- [ ] 出典・由来が分かる
- [ ] AIが何をしたかが分かる
- [ ] frontmatterだけでは分からない判断理由や迷いが本文に書かれている
- [ ] 古いノートや関連ノートがあればリンク候補が書かれている
- [ ] 結論後に、必要に応じて3行程度の「次に残る問い」を置いている

## 読みやすさ

- [ ] 冒頭が定義や要約だけでなく、具体的な観察から始まっている
- [ ] 事例は3〜5個程度に絞られ、本文の主張に効いている
- [ ] 各事例に「場所」「使われ方」「何が測定・分類・誘導されるか」がある
- [ ] 必要に応じて、企業名、自治体名、施設種別、年、数値などの確認済み具体例が短く入っている
- [ ] 数字や導入効果を入れる場合、企業発表・報道・公的資料などの出典に基づいている
- [ ] 出典説明が本文を圧迫していない
- [ ] 参考・出典は最重要なものだけに絞り、リンク総数で5〜6個に収めている
- [ ] 参考・出典の説明はリンクの横ではなく、リンク下に短く置いている
- [ ] 専門用語が多い記事では、必要な章末に短い用語メモがある
- [ ] 用語解説はその記事を読むために必要な意味だけに絞っている
- [ ] 同じ説明を繰り返していない
- [ ] 最後に、事例を束ねる共通構造が短く整理されている
- [ ] 記事を確定知識のように閉じず、未確認点や結論の射程を短く残している
- [ ] 批判はしていても、煽りや断罪調になっていない
- [ ] 画像を使う場合、著作者、出典URL、ライセンス名、ライセンスURL、変更の有無が分かる

## 公開判断

- [ ] 個人情報、APIキー、非公開情報が入っていない
- [ ] 出典を読んでいないのに、読んだように書いていない
- [ ] 強すぎる断定をしていない
- [ ] 迷う場合は `status: seed` にしている
- [ ] 人間が公開してよいと判断している

## サイドバー

- [ ] 新しい記事を既存の Starlight サイドバーカテゴリに入れるべきか確認した
- [ ] 既存カテゴリに入れる場合、カテゴリ名と理由を最終報告に書く
- [ ] 新しいカテゴリが必要そうな場合、勝手に追加せず、候補名と理由を提案する
- [ ] すでにユーザーがカテゴリを指定している場合、その指定に従った

## PRに書くとよいこと

- 追加・変更した記事
- `status` とその理由
- 主な出典・由来
- AIが手伝ったこと
- `confidence` と `review_needed`
- まだ確認したいこと
- 関連ノートや内部リンクの候補
- サイドバーに入れるカテゴリ案
